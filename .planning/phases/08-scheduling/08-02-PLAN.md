**Step 1: Verify Azure Container Apps min-replicas**

Check the current min-replicas setting for the n8n container app:
```bash
az containerapp show --name <app-name> --resource-group <rg> --query "properties.template.scale.minReplicas"
```

If min-replicas is 0, update to 1:
```bash
az containerapp update --name <app-name> --resource-group <rg> --min-replicas 1
```

The app name and resource group should be found from the Azure CLI context or .env. If not found, check `az containerapp list` to discover the n8n container app.

This is a BLOCKING prerequisite — do NOT proceed to deploy or test until min-replicas=1 is confirmed. Scale-to-zero kills Wait node executions silently (RESEARCH.md Pitfall 6).

**Step 2: Add publish_at column to Supabase content_sessions**

Before deploying, ensure the Supabase table has the new column. Use the Supabase Management API or SQL:
```sql
ALTER TABLE content_sessions ADD COLUMN IF NOT EXISTS publish_at TEXT DEFAULT 'now';
```

Execute via Supabase REST API (same pattern as Phase 7 schema extension) or ask user to run in Supabase SQL editor if API access is not available.

**Step 3: Deploy workflow to n8n-azure**

Use the n8n API to update the workflow (same pattern as Phase 7 Plan 03):
1. Read n8n/workflow.json
2. Strip the `settings` object down to only `executionOrder` (full settings cause 400 error — Phase 7 decision)
3. PUT to n8n API: `PUT /api/v1/workflows/{WORKFLOW_ID}`
4. Activate the workflow if not already active

The workflow ID and n8n API base URL should be extracted from existing deployment scripts or .env. Check Phase 7 Plan 03 SUMMARY for the exact deployment pattern used.
