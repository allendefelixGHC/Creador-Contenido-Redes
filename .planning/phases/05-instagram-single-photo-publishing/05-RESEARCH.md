## Summary

Phase 5 extends the main workflow with a 4-node Meta publish chain inserted between `🔗 Merge Rehost Output` and `📊 Google Sheets Log`: (1) HTTP POST to create the IG media container, (2) a 30-second Wait node for container readiness, (3) HTTP POST to `media_publish` with retry disabled, (4) HTTP GET to fetch the `permalink`. A 5th node (WhatsApp success notification via HTTP Request to YCloud) runs in parallel with or before the Sheets Log update, and the Sheets Log node schema is extended to include the new `IG_URL`, `Publicado_En`, and `Publish_Status` columns.

The critical risk in this phase is **duplicate posts on the live Instagram profile**. `media_publish` is not idempotent — if n8n times out before receiving the response but Meta has already accepted the publish, a retry will create a second live post (Meta has no documented server-side dedupe). The only defense is disabling retry on the `media_publish` node specifically. Container creation (step 1) IS idempotent and SHOULD have retry enabled (ERR-01, deferred to Phase 9 but safe to enable here). The Wait node gives the container time to transition from `IN_PROGRESS` to `FINISHED`; without it, `media_publish` fails with error `9007/2207027` ("The media is not ready for publishing").

**Primary recommendation:** Insert 4 new HTTP Request nodes + 1 Wait node between `🔗 Merge Rehost Output` and `📊 Google Sheets Log`, using `graph.instagram.com/v22.0` as the host and Susana's Page Access Token via the existing `META_PAGE_TOKEN` env var. Add WhatsApp success notification as a separate HTTP Request node after permalink fetch. Update Google Sheets node `columns.value` in place — no Sheets schema migration needed in code; the user adds columns in the Sheet UI. Do NOT add carousel branching in Phase 5 — Phase 7 will refactor with its own IF node; premature branching now adds dead code.

## Per-Question Research Answers

### Q1: IG Graph API v22 single-photo publish flow — exact endpoints

**Confidence:** HIGH (official Meta docs).

#### Step 1 — Create Media Container

```
POST https://graph.instagram.com/v22.0/{INSTAGRAM_ACCOUNT_ID}/media
Content-Type: application/json
```

Request body (JSON, but Meta also accepts query-string — stick to JSON):
```json
{
  "image_url": "https://propulsarcontent.blob.core.windows.net/posts/2026/04/16/<uuid>.jpg",
  "caption": "<text from $json.instagram — 2200 char max>",
  "access_token": "{{ $env.META_PAGE_TOKEN }}"
}
```

Response:
```json
{ "id": "17841400000000000" }
```
This is the **container ID** (creation_id for step 2).

**Required:** `image_url`, `access_token`.
**Optional but useful:** `caption` (max 2,200 chars, **includes** hashtags and URLs in the count), `alt_text` (max 1,000 chars, accessibility), `user_tags`, `location_id`.
**Image requirements:** JPEG only, max 8 MB, aspect ratio 4:5 to 1.91:1, sRGB color space.

> Sources: [Meta — Create Media Container](https://developers.facebook.com/docs/instagram-platform/instagram-graph-api/reference/ig-user/media/) · [Meta — Content Publishing Flow](https://developers.facebook.com/docs/instagram-platform/content-publishing/)

#### Step 2 — Publish Container

```
POST https://graph.instagram.com/v22.0/{INSTAGRAM_ACCOUNT_ID}/media_publish
Content-Type: application/json
```

Body:
```json
{
  "creation_id": "{{ $json.id }}",
  "access_token": "{{ $env.META_PAGE_TOKEN }}"
}
```

Response:
```json
{ "id": "17841400000000001" }
```
This is the **published media ID** (used to fetch permalink in step 4).

> Source: [Meta — media_publish reference](https://developers.facebook.com/docs/instagram-platform/instagram-graph-api/reference/ig-user/media_publish/)

#### Caption field handling (IMPORTANT)

- **Max length:** 2,200 characters (hashtags + URLs included in count).
- **Only first 125 chars visible** in feed before the "more" tap — front-load the hook.
- Supports `@mentions` and `#hashtags` inline, but per **IGPUB-06 (Phase 9 deferred)**, hashtags should eventually move to a first comment. For Phase 5, caption = `$json.instagram` as-is from GPT-4o.

### Q2: Container readiness — confirm 30-second Wait

**Confidence:** HIGH (matches Meta's guidance + community consensus).

For **single-photo** posts, 30 seconds is overkill in most cases (single-image containers typically hit `FINISHED` in 2-5 seconds) but is the safe industry-standard fixed wait. Meta's official recommendation is "poll once per minute for up to 5 minutes" — since we're explicitly choosing Wait over polling (HARD-03 defers polling to v2), 30s is the conservative-but-fast choice.

**If we publish too early:** Error `9007` subcode `2207027`:
> "The media is not ready for publishing, please wait for a moment"

This is a retryable error, but the cleanest defense is simply waiting long enough up front. 30 seconds gives a large safety margin for single photos.

**Carousel note (out of scope Phase 5):** Carousels need 30s+ because each child container has its own processing. Phase 5's 30s pattern ports cleanly to Phase 7.

> Sources: [Meta — Content Publishing](https://developers.facebook.com/docs/instagram-platform/content-publishing/) · [Meta — Error Codes](https://developers.facebook.com/docs/instagram-platform/instagram-graph-api/reference/error-codes/) · [IG Container reference](https://developers.facebook.com/docs/instagram-platform/instagram-graph-api/reference/ig-container/)

### Q3: Permalink retrieval

**Confidence:** HIGH (official docs + verified working endpoint).

After `media_publish` returns `{ id: "<media_id>" }`:

```
GET https://graph.instagram.com/v22.0/{media_id}?fields=permalink&access_token={{ $env.META_PAGE_TOKEN }}
```

Response:
```json
{
  "id": "17841400000000001",
  "permalink": "https://www.instagram.com/p/XXXXXXX/"
}
```

**Caveats:**
- **Works on top-level posts only.** The `permalink` field is NOT available on carousel children (irrelevant for Phase 5 — single posts only).
- **Token permission:** Long-lived Page Access Token (Susana's) with `instagram_basic` + `instagram_content_publish` scopes covers this. Already confirmed working per STATE.md.
- **Timing:** Permalink is available immediately after `media_publish` returns success — no additional wait needed.
- **Optional extra fields:** Can request `id,permalink,timestamp,caption,media_type` in one GET to reduce round-trips if we ever want to store more metadata.

> Source: [Meta — IG Media reference](https://developers.facebook.com/docs/instagram-platform/reference/instagram-media/)

### Q4: n8n node choices + retry config

**Confidence:** HIGH (verified from project history + official n8n docs).

#### For each Meta call

| Step | Node type | typeVersion | Retry on Fail | On Error |
|------|-----------|-------------|---------------|----------|
| 1. Create container | `n8n-nodes-base.httpRequest` | 4.2 | **ENABLED** (3 tries, 2000ms) — idempotent | `stopWorkflow` (Phase 9 adds branch) |
| 2. Wait 30s | `n8n-nodes-base.wait` | 1 | N/A | N/A |
| 3. media_publish | `n8n-nodes-base.httpRequest` | 4.2 | **DISABLED** — NOT idempotent (CRITICAL) | `stopWorkflow` |
| 4. GET permalink | `n8n-nodes-base.httpRequest` | 4.2 | ENABLED (3 tries, 1000ms) — idempotent | `stopWorkflow` |
| 5. WA notify success | `n8n-nodes-base.httpRequest` | 4.2 | ENABLED (2 tries) — idempotent | `stopWorkflow` |

#### How to DISABLE retry on `media_publish` node (CRITICAL)

In the node JSON:
```json
{
  "parameters": { ... },
  "retryOnFail": false,          // MUST be false (or omitted — false is default)
  "maxTries": 1,                 // Safety belt — 1 means no retry
  "waitBetweenTries": 0,
  "onError": "stopWorkflow"      // No error branch in Phase 5; Phase 9 changes to continueErrorOutput
}
```

In the n8n UI: Settings tab → "Retry on Fail" toggle **OFF**.

**Why `onError: stopWorkflow` (not `continueErrorOutput`) in Phase 5:**
- Phase 5's scope per REQUIREMENTS.md covers only `ERR-02` (retry disabled), not `ERR-04` (error output branch).
- Full error-output branching on all Meta nodes is Phase 9.
- Keeping `stopWorkflow` in Phase 5 means a `media_publish` failure aborts the execution cleanly — no duplicate post risk, error visible in n8n execution log, Felix can check manually. WhatsApp failure UX waits for Phase 9.
- **Known n8n 2.14.2 bug:** `continueErrorOutput` on HTTP Request sometimes swallows errors with `"Cannot read properties of undefined (reading 'error')"` ([GitHub #18113](https://github.com/n8n-io/n8n/issues/18113)) — another reason to defer error branching to Phase 9 when the UX matters.

#### Wire the new chain

Current:
```
🔗 Merge Rehost Output → 📊 Google Sheets Log
```

Phase 5 becomes:
```
🔗 Merge Rehost Output
   → 📤 IG: Create Container (HTTP Request 4.2, retryOnFail=true)
   → ⏳ Wait 30s (wait node)
   → 🚀 IG: media_publish (HTTP Request 4.2, retryOnFail=FALSE)   ← CRITICAL
   → 🔗 IG: Get Permalink (HTTP Request 4.2, retryOnFail=true)
   → ✅ Notify WhatsApp Success (HTTP Request 4.2, retryOnFail=true)
   → 📊 Google Sheets Log (existing — update columns)
```

> Sources: [n8n — Error handling](https://docs.n8n.io/flow-logic/error-handling/) · [n8n — HTTP Request common issues](https://docs.n8n.io/integrations/builtin/core-nodes/n8n-nodes-base.httprequest/common-issues/) · Project STATE.md

### Q6: Google Sheets update pattern

**Confidence:** HIGH (n8n docs) — MEDIUM (project's historical choice).

**Current state:** The existing `📊 Google Sheets Log` node uses `operation: append` (implicitly — only adds new rows). Columns currently in use: `Fecha`, `Tema`, `Tipo`, `Angulo`, `Plataformas`, `Modelo_Imagen`, `Imagen_URL`, `Estado`.

**Phase 5 needs:** Add `IG_URL`, `Publicado_En`, `Publish_Status` columns (LOG-01, LOG-02).

#### Recommendation: Keep `append`, don't switch to `appendOrUpdate`

**Reasoning:**
1. The existing Supabase session table has `session_id` but **no matching column in the Sheet yet**. `appendOrUpdate` requires a matching key column.
2. Each approval flow produces exactly ONE Sheet row — there's nothing to update across sessions. Append is simpler and matches current project state.
3. LOG-02 says "actualiza O appendea" — either is compliant.
4. Switching to `appendOrUpdate` requires adding a `session_id` / `post_id` column to the Sheet AND backfilling historical rows, which is scope creep.

**Caveat:** If Felix later wants idempotent re-runs (same `post_id` → same row), switch to `appendOrUpdate` with `post_id` as the key column in Phase 9. Flag for future consideration.

#### Exact column update in the existing node

Update the `columns.value` object in `📊 Google Sheets Log` node:

```json
{
  "Fecha": "={{ new Date().toISOString() }}",
  "Tema": "={{ $('🔗 Merge Rehost Output').item.json.topic }}",
  "Tipo": "={{ $('🔗 Merge Rehost Output').item.json.type }}",
  "Angulo": "={{ $('🔗 Merge Rehost Output').item.json.angle || '' }}",
  "Plataformas": "={{ Array.isArray($('🔗 Merge Rehost Output').item.json.platforms) ? $('🔗 Merge Rehost Output').item.json.platforms.join(', ') : '' }}",
  "Modelo_Imagen": "={{ $('🔗 Merge Rehost Output').item.json.image_model }}",
  "Imagen_URL": "={{ ($('🔗 Merge Rehost Output').item.json.blob_urls && $('🔗 Merge Rehost Output').item.json.blob_urls[0] && $('🔗 Merge Rehost Output').item.json.blob_urls[0].url) || '' }}",
  "IG_URL": "={{ $('🔗 IG: Get Permalink').item.json.permalink }}",
  "Publicado_En": "={{ new Date().toISOString() }}",
  "Publish_Status": "success",
  "Estado": "Publicado"
}
```

**Key change:** All topic/type/angle references now go through `$('🔗 Merge Rehost Output').item.json.*` instead of bare `$json.*` — because the incoming item at Sheets Log is now the output of the permalink fetch (i.e., `{ id, permalink }`), NOT the merged session data. This is an application of the **Set v3 cross-ref lesson** from STATE.md: Code-node/expression cross-refs MUST use `$('node-name').item.json.*` form when fan-out/step chains change what `$json` means.

**Manual step:** Felix must add `IG_URL`, `Publicado_En`, `Publish_Status` columns to the Google Sheet UI before Phase 5 test runs. Cannot be done in code.

### Q9: Project-specific pitfalls

**Confidence:** HIGH (all from STATE.md + memory files + git history).

- **Context:** All new IF nodes must use typeVersion 1.
- **Impact on Phase 5:** Per Q8, no new IF nodes needed. If Phase 5 ever adds one (e.g., for the optional carousel guard), use `typeVersion: 1`.

#### Pitfall 3: `N8N_BLOCK_ENV_ACCESS_IN_NODE=true` on the Container App

- **How to avoid:** Don't add Code nodes in Phase 5 that reference env vars. The existing `🔧 Prep Re-host Input` Code node already avoids this.

#### Pitfall 4: n8n Code node sandbox disallows `require('crypto')`
- **Context:** Bug patched in Phase 4. Sub-workflow now uses Math.random UUID v4.

## Meta Graph API Error Codes Reference

Map Phase 5 failure modes to specific Meta error codes:

| Error Code / Subcode | Meaning | When it happens | Phase 5 handling |
|----------------------|---------|-----------------|------------------|
| `9007` / `2207027` | Media not ready for publishing | `media_publish` called before container reaches `FINISHED` | Increase Wait beyond 30s; log says "wait a moment" — Phase 5 aborts, Felix checks |
| `9004` / `2207052` | Media could not be fetched from URI | Azure Blob URL unreachable (CDN hiccup, wrong URL, private blob) | Container creation fails — retry 3x catches transient; permanent failure aborts, Felix verifies blob URL in browser |
| `9` / `2207042` | Max posts reached (50 per 24h) | Shouldn't happen for Propulsar unless stress-testing | Aborts execution; Felix waits or checks IG insights |
| `190` | OAuthException — token expired/invalid | Susana lost admin role, or token expired | Phase 9 adds specific alert; Phase 5 just aborts |
| `2` (null subcode) | Generic retryable unknown error | Rare, transient | Container creation retry covers; `media_publish` does NOT retry (safety > reliability) |

## Common Pitfalls (Summary)

### Pitfall: Enabling retry on media_publish
**What goes wrong:** Duplicate live post on Instagram profile.
**Why:** `media_publish` is not idempotent. If Meta processes the request but n8n times out before receiving `200 OK`, retry creates a second live post.
**How to avoid:** `retryOnFail: false` on THAT node specifically. Set `maxTries: 1` as safety belt.
**Warning signs:** Two posts with the same caption appear on `instagram.com/propulsar.ai` with timestamps seconds apart.

### Pitfall: Using `$json` at Google Sheets node for topic/type/angle
**What goes wrong:** Sheet row has blank `Tema`, `Tipo`, `Angulo` columns.
**Why:** The item at Google Sheets Log is now the output of `🔗 IG: Get Permalink` (`{ id, permalink }`), NOT the merged session data. `$json.topic` is undefined.
**How to avoid:** Use `$('🔗 Merge Rehost Output').item.json.topic` cross-node references.
**Warning signs:** Blank columns in Sheet, no errors thrown.

### Pitfall: Hard-coded double quotes in YCloud jsonBody
**What goes wrong:** YCloud returns 400 "Invalid JSON".
**Why:** Manually writing `'{"from": "..."}'` with quotes inside expressions creates escaping issues (previously fixed in commit `effab36`).
**How to avoid:** Always use `{{ JSON.stringify({...}) }}` wrapper — matches project's established pattern.
**Warning signs:** YCloud HTTP 400 with "malformed JSON" message.

### Pitfall: Not verifying Meta token via test GET before running publish
**What goes wrong:** Expensive full-flow test fails at media_publish with 190.
**Why:** Token expired/invalid; no cheap dry-run.
**How to avoid:** Before running Phase 5 first end-to-end test, run a manual `curl` GET against `https://graph.instagram.com/v22.0/{IG_ID}?fields=id&access_token={TOKEN}`. If this returns `{ id, ... }`, token works. If 190, refresh token first.

### Pitfall: Adding columns to the Sheet AFTER the first failed run
**What goes wrong:** First run appends to wrong columns; misaligned data.
**Why:** Google Sheets node matches by column name — if column doesn't exist, it silently appends a new column header and shifts.
**How to avoid:** Felix adds the 3 new columns in Sheet UI BEFORE Phase 5 first test, exactly in order: `IG_URL`, `Publicado_En`, `Publish_Status` — appended to the right of existing columns.

## Open Questions

1. **Should Phase 5 pre-emptively enable retry on container creation (ERR-01)?**
   - What we know: ERR-01 is Phase 9 per REQUIREMENTS.md traceability.
   - What's unclear: Whether enabling it in Phase 5 counts as "scope creep" or "safe default".
   - Recommendation: Enable it now. Retry on an idempotent operation is a safe default, not a feature. If the planner prefers strict traceability, set `retryOnFail: false` on container creation too and let Phase 9 enable it — but document the choice.

2. **Should the WhatsApp success node run BEFORE or AFTER Google Sheets Log?**
   - What we know: Phase 5 requires both (NOTIF-01, LOG-02).
   - What's unclear: Order matters for failure modes — if Sheets Log fails, does WA still send? If WA fails, does Sheets still log?
   - Recommendation: **WA before Sheets.** Felix wants the live confirmation on his phone ASAP. Sheets logging is for audit and can tolerate a few seconds delay. If Sheets fails after WA succeeds, Felix can manually re-add the row from the WA message data.

3. **Should Phase 5 expose the `media_id` or `container_id` anywhere for debugging?**
   - What we know: The permalink is sufficient for the user-facing message.
   - What's unclear: If a post's permalink looks wrong, can Felix diagnose?
   - Recommendation: Log `media_id` and `container_id` to the Google Sheet as a debug column (`IG_Media_ID`). Optional. Flag for user decision — if they prefer a minimal Sheet, skip it.

4. **Is the Meta Page Token available to n8n as an env var right now?**
   - What we know: STATE.md says Meta tokens resolved 2026-04-10 from Susana's account.
   - What's unclear: Whether it's in the n8n Container App env vars yet OR just stored in CLAUDE.md context.
   - Recommendation: **Planner must verify** via Azure CLI: `az containerapp show ... | grep META_PAGE_TOKEN`. If not present, planner adds a prep task: push the token to the Container App env BEFORE implementing Phase 5 nodes.

5. **Does the IG token use `graph.instagram.com` or `graph.facebook.com`?**
   - What we know: Both hosts work for the same endpoints; `graph.instagram.com` is the IG-Login-specific host, `graph.facebook.com` is the Facebook-Login-for-Business host.
   - What's unclear: Which login flow was used to generate Susana's long-lived Page Access Token.
   - Recommendation: Test both with a cheap `curl` GET before committing node config. Default to `graph.instagram.com` (matches Meta's current examples) and fallback to `graph.facebook.com` if token rejects it. Planner should call this out as a checkpoint.
